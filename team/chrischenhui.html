1. Overview of Dukemon

Figure 1. The Home Page of Dukemon upon initial start-up.

Dukemon aims to streamline and gamify the process of learning words or definitions through the use of self-created digital flashcards.

These flashcards are termed as Cards.
Every Card is contained in a WordBank.

General flow of program usage:

  • Note taking phase

    1. User creates/selects a WordBank

    2. User populates/edits the WordBank with Cards

  • Learning phase

    1. User starts the game using selected WordBank

    2. Dukemon prompts questions, and user has to answer

    3. Dukemon ends the game

    4. User reviews performance from statistics page

2. Summary of contributions

2.1. Major feature

Word Bank Management.

  • Feature:

    • Enables user to have full control over his word banks and cards through typed commands and graphical user interface. Behind the scenes, Dukemon manages your word banks automatically.

      1. Word Bank’s data structure and storage:
        Is implemented with a robust underlying WordBankList data structure, with it’s own WordBankListStorage, along with rich integration of Model, Storage, Logic and UI to synchronise it’s data.

      2. User Commands:
        Allows users to edit cards/word banks through CardCommands/ WordBankCommands.

      3. Drag and Drop:
        Supports sharing of word banks with friends.
        User can export their word bank to their computer simply by dragging it out of Dukemon.
        Likewise, drag a word bank file into Dukemon to import the word bank.

      4. Revision Word Bank:
        Serves as a centralised bank that collects cards across different word banks throughout the game play.
        Cards that were answered wrongly are automatically added to this revision bank.
        Likewise, cards that were answered correctly during game play are automatically removed from this revision bank.

  • Justification:

    • Allowing user to edit his own card and grouping these cards according to topics (Word Banks) is essential for a self-typed flashcard app like Dukemon.

    • Well designed architecture and well implemented data structure streamlines integration with other features that builds on top of it. (statistics, settings, game modes)

    • Revision bank improves learning experience for the user, by helping the user automatically aggregate cards that were answered wrongly.

    • Learning can be enhanced through sharing of knowledge/notes with friends. Drag and Drop feature accelerates this process.

  • Highlights:

    • As this feature lays the basis for the app, it needs to be implemented well to support integration with every component of the app.

5.1. Model component

Figure 2. Structure of the Model Component

API : Model.java

The Model,

  • contains information that the game requires at run time. They include: WordBankList, WordBankStatisticsList, GlobalStatistics, Game, AppSettings, UserPrefs.

  • does not depend on any of the higher level components. i.e. Ui, Timer, AppManager, Logic, Storage

  • has a direct reference to a user selected WordBank.

  • exposes an unmodifiable ObservableList<Card> that can be 'observed' e.g. the UI can be bound to this list so that the UI automatically updates when the data in the list change.

  • can be visualised as playing the role of a computer’s RAM, while Logic plays the role of an operating system, and Storage plays the role of a hard disk.

5.2. Storage component

Figure 3. Structure of the Storage Component

API : Storage.java

The Storage component,

  • contains multiple types of distinct storage.

  • handles function calls directly to the computer’s system.

  • can save data objects in json format and read it back.

  • can be visualised as playing the role of a computer’s hard disk, while Logic plays the role of an operating system , and Model plays the role of a RAM.
